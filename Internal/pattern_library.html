Foundation P01
Methodology spine encoding
QiyasOS · design CutoverOS · built TransformationOS · concept
The structured framework, perspectives, axes, standards, scoring rubric, lifecycle, becomes the spine of the platform. Every screen, question, gap, score, and briefing traces back to a framework element. Not glued on; built in.
The spine isn't referenced, it's the runtime. Same five-segment shape; three different anchor types. QiyasOS anchors to static framework standards. TransformationOS anchors to dynamic programme objects (actions, issues, decisions, change requests), the spine grows from signal. CutoverOS anchors to a time-bounded cutover plan. One engine, three engagement cadences.
How it works
The framework loads once as schema, dimensions × components × atomic assessables × rubric × weights. UI generators, prompts, rubrics, and exports all read from that schema. Swapping frameworks means a new schema, not a new engine, QiyasOS (10×23×95) and D365 F&O (38×5×~6,800) run the same engine under the hood.
Why it matters
A generic AI tool answers questions about a framework; a spine-encoded engine reasons within it. Every other pattern in this library inherits the spine, so the moat compounds across products, not just within one.

Capture P02
Multi-source signal ingestion
QiyasOS · design CutoverOS · built TransformationOS · concept
Ingest any format (docs, email, Teams transcripts, voice, handwritten) in any language. Each source type routes to a specialist parser. Outputs are mapped to whatever the engagement's spine is, Qiyas framework standards, TransformationOS programme objects (actions, issues, decisions), or CutoverOS tasks, with citation, confidence, and operation (LINK / NEW / UPDATE). No manual indexing, no type-blind ingestion.
Type-aware ingestion · engagement-anchored output. The same five parsers feed three different spine shapes, a Qiyas framework standard, a TransformationOS programme object (where claims can create or update actions, decisions, issues), or a CutoverOS task. The pattern is identical; the engagement decides what the anchor is.
How it works
A normalisation layer (OCR · STT · parsers · telemetry decoders) turns each source type into structured claims with metadata. A type-aware router sends each to its specialist parser. Vector similarity + LLM verification map claims to framework elements with confidence. All citations stay anchored to page · timestamp · row · scan location.
Why it matters
Real engagements run on heterogeneous evidence. Manual indexing kills cycle time; missing the long tail biases scoring. A type-blind generic ingestion treats every source as text and loses what makes each format useful. Page-level citation is non-negotiable for audit defensibility downstream.

Capture P03
Gap-driven interview design
QiyasOS · design CutoverOS · built TransformationOS · concept
Derive the interview set from what documents leave behind. Pre-answer what's evidenced; ask only what's truly open. Every stakeholder slot is the minimum needed, routed to the person who owns that area (the funding gap goes to the CFO, the training gap to the CHRO), not whoever happens to be most senior.
The pool is fixed; the interview shrinks. Most questions are already answered by the evidence base, so only the open ones need stakeholder time. Each open slot routes to whoever owns that area (a funding gap goes to the CFO, a training gap to the CHRO), not whoever happens to be most senior.
How it works
After P2 (Multi-source signal ingestion), every framework standard is tagged evidenced / partial-low-confidence / unevidenced. The question generator drafts interview slots only for the latter two, picks the stakeholder closest to the gap, and orders questions by closure-value.
Why it matters
Stakeholder time is the scarcest engagement resource. Asking what docs already answer wastes the stakeholder and the consultant, and worse, it signals you didn't read the room. Pre-answered slots earn trust.

Capture P04
Dual-mode stakeholder engagement
QiyasOS · design CutoverOS · concept TransformationOS · concept
Full AI autonomous or consultant-led, chosen per stakeholder, not per engagement. Both modes write to one shared transcript schema and one shared flag log, so the downstream conflict synthesis (P5) reads either input identically.
Two modes, one output schema. The C-suite refuses to talk to a bot. Many ops leads can be interviewed by AI in parallel on their own schedule, which consultant-led can't match. Per-stakeholder mode choice; both modes write to the same transcript schema, so P5 (Cross-source conflict synthesis) downstream cannot tell which mode produced it.
How it works
Both modes write to the same transcript schema. In consultant-led mode the AI co-pilot listens and flags contradictions live; in autonomous mode the AI runs the conversation and exposes the same flags for consultant review. Output schema identical → P5 (Cross-source conflict synthesis) doesn't care.
Why it matters
Different stakeholders demand different modes. The C-suite won't sit with an AI. AI can run interviews with dozens of ops leads in parallel on their own schedule, which consultant-led can't match. A single mode forces compromise; two modes with shared output downstream don't. The choice is per-person, not per-engagement.

Analysis P05
Cross-source conflict synthesis
QiyasOS · design CutoverOS · built TransformationOS · concept
Inconsistencies, doc vs doc, doc vs interview, claim vs evidence, surface as discrete conflict cards with severity, named owners, and structured resolution paths. Not buried in narrative; promoted to first-class objects. (P11 (Cross-dimension dependency modelling) handles the *structural* class, score-vs-dependency conflicts, separately.)
Conflicts are first-class objects, not paragraphs. Every contradiction between sources pinned to the same anchor surfaces as a discrete card with severity, owner, and a resolution-path menu. The owner picks A / B / C; the choice carries forward into the audit trail managed by P10 (Audit-traceable continuous evidence base).
How it works
All claims (from P2 (Multi-source signal ingestion) and P4 (Dual-mode stakeholder engagement)) are indexed against framework elements. The contradiction detector pairs claims, scores conflict severity, names the owner closest to authority, and proposes resolution paths drawn from a structured catalogue.
Why it matters
In every engagement, most "gaps" are inconsistencies you missed, not missing evidence. Surfacing them as cards (not paragraphs) changes who owns and when. CutoverOS surfaced 47 conflicts and resolved 39 live; without cards, those die in slides. (Structural score-vs-dependency conflicts are P11's (Cross-dimension dependency modelling) job, separate pattern, same first-class treatment.)

Analysis P06
Benchmark-validated estimates
QiyasOS · concept CutoverOS · built TransformationOS · concept
Every duration, effort, and target is cross-checked against prior instances, not asserted from one expert's last engagement. Estimates carry cohort-anchored mid-estimates and honest sensitivity bands.
The expert estimate is one data point. The cohort is fourteen. Mid-estimate anchored to the matched cohort distribution; sensitivity band derived from cohort spread, not from confidence. When the client pushes back, the cohort is the answer.
How it works
A library of prior instances is indexed by framework element, dimension, and entity profile. New estimates anchor to the matched cohort distribution. Sensitivity bands derive from the spread in that cohort, not consultant gut-feel.
Why it matters
Single-expert estimates are systematically optimistic in transformation work. Cohort anchoring gives both a defensible mid-estimate and a sensitivity band, both derived from the same peer data. Clients can ask "where does that range come from?" and the answer is "these 14 prior instances", which is the only honest way to defend an estimate under challenge.

Analysis P07
Maturity scoring with peer benchmark
QiyasOS · design CutoverOS · concept TransformationOS · concept
Score against the framework with confidence intervals; benchmark against anonymised peers in real time. Targets per dimension are set against baseline, priority, and realistic in-cycle improvement, not a flat uplift. Confidence binds explicitly to evidence-marker coverage at each level.
The score, the confidence, and the target, all from the same engine. Each level's evidence markers are counted and verified individually; the confidence band is literally the marker-coverage rate. Targets are the realistic next percentile band, not a flat uplift.
How it works
P1 (Methodology spine encoding) provides the rubric. P2 (Multi-source signal ingestion) supplies the evidence (each claim carries its own confidence from the parser). P5 (Cross-source conflict synthesis) reconciles conflicting claims first; the survivors feed the score. Confidence at each level = fraction of that level's evidence markers verified. An anonymised cohort produces percentile bands per dimension. A target generator proposes per-dimension targets sensitive to baseline + priority + cycle length.
Why it matters
Flat-uplift targets are politically safe but technically wrong, dimensions move at different speeds. Peer percentiles ground the conversation in achievable, not aspirational. Confidence intervals let the reader tell signal from noise.

Synthesis P08
Plain-English consequence mapping
QiyasOS · design CutoverOS · built TransformationOS · concept
Every maturity framework includes a generic "why this matters" paragraph for each dimension (e.g. "less-than-monthly risk reviews reduce executive visibility into emerging threats"). P8 (Plain-English consequence mapping) takes that generic statement and quantifies it for this entity, this region, this cycle (turning it into "SAR 4.8M annual risk exposure for a 42k-staff entity serving 3.2M citizens"). Every gap and every initiative ends up with a € / SAR / FTE / audit-timing / citizen-impact consequence in plain language, board- and exec-ready, not audit-jargon.
Generic in, specific out. The framework writes "leadership visibility into emerging risks." The engine writes "SAR 4.8M annual exposure for a 42k-staff entity serving 3.2M citizens." Same fact, board-actionable.
How it works
A consequence catalogue is mapped per framework element. A cost-of-inaction model is parameterised by entity size, region, sector. A plain-language renderer composes one paragraph per gap or initiative, same shape every time so readers learn to scan it.
Why it matters
Gaps without consequence framing don't get prioritised. Boards approve what they can sign their name to in plain language, not what reads like an audit finding. The plain-English layer is what makes the rest of the engine board-actionable.

Synthesis P09
Gap → initiative → roadmap
QiyasOS · design CutoverOS · built TransformationOS · concept
Translate every gap into a named initiative with sized effort, dependency, owner, and consequence framing. Co-design the roadmap with the client in the platform, so every move (accept, defer, re-sequence, reject) is logged against the evidence behind it. The output is a signed plan whose audit trail is intact, not a slide deck typed up after a workshop with the link to the underlying discussion lost.
Gaps to a signed plan, in-platform. The framework's initiative catalogue (~3,200 indexed for D365 Compass alone) supplies the candidates; P6 (Benchmark-validated estimates) sizes them; P11 (Cross-dimension dependency modelling) sequences them. The client moves, resizes (extends or shortens), defers, or accepts each initiative in the same view; every change becomes part of the audit trail.
How it works
Each gap proposes 1–3 named initiatives from an indexed catalogue, sized via P6 (Benchmark-validated estimates), framed via P8 (Plain-English consequence mapping). The roadmap composer sequences them by dependency + capacity. A co-design panel lets the client move, resize, defer, or accept; every change is logged and signed.
Why it matters
Gap reports without owned initiatives die on the shelf. Initiatives without dependency sequencing produce un-executable roadmaps. Co-design in-platform is what turns a deliverable into a signed plan with an audit trail.

Analysis P11
Cross-dimension dependency modelling
QiyasOS · concept CutoverOS · concept TransformationOS · concept
Maturity frameworks list dimensions side-by-side, as if they're independent. They're not. Funding depends on Business Case; Hypercare depends on Testing and Training. P11 (Cross-dimension dependency modelling) encodes the implicit dependency graph framework authors leave out, and uses it to flag structurally implausible scores, sequence roadmaps causally, and prioritise continuous evidence on upstream dimensions.
Framework authors leave dimensions side-by-side. They aren't. P11 (Cross-dimension dependency modelling) encodes the implicit dependency relationships and turns "a downstream score is higher than its upstream dependency allows" into a first-class structural conflict. Feeds P9 (Gap → initiative → roadmap) with causal sequencing (fix upstream first), not just severity-ordered work.
How it works
On framework load, P11 (Cross-dimension dependency modelling) reads (or authors) dependency relationships between dimensions — each one is a sentence like "D5 Funding depends on D2 Business Case". These come from expert encoding or are inferred from cohort patterns (whenever peers' D5 was low, D2 was also low → infer the dependency). At scoring time, every dimension's score is checked against the dimensions it depends on. Violations — like "Funding is Strong but its upstream Business Case is Developing" — surface as structural-conflict cards.
Why it matters
A framework with 38 dimensions has implicit dependencies between many of them, but authors don't formalise them. Without the dependency graph, scores look additive; readers can't tell that "Funding Strong + Business Case Developing" is implausible. With it, two things happen automatically. (1) Implausible score combinations are flagged before publication. (2) P9 (Gap → initiative → roadmap) sequences the work causally — for this example, the roadmap recommends "fix Business Case first, then strengthen Funding", not "optimise both in parallel".

Anti-pattern (the framework's own omission)
The framework presents dimensions as if independent and aggregates scores additively. A client claims sponsorship Strong + decision-effectiveness Developing + funding Strong; the average looks reasonable; the underlying combination is structurally implausible. Without P11 (Cross-dimension dependency modelling), the engine can't see this, and the client signs off on a roadmap that sequences initiatives in the wrong order.
